# 全电发票（数电票）开票方案选型对比

> 面向：思科达牙科仪器售后维修小程序 ｜ 决策人：财务 / 负责人
> 目的：为「签收后申请发票」的后台一键开票能力选定对接方案
> 结论先行：**推荐「第三方 SaaS（诺诺优先）」**，理由见文末。

## 一、背景与现状

- 客户端「申请开票」已完整（抬头/税号/邮箱采集、四阶段状态流），后台发票状态字段也就绪。
- **唯一缺口**：后台「一键开票」是桩 —— `cicada-admin-order` 的 `callInvoiceProvider()` 直接抛「开票接口待对接」。
- 现有适配层 `getInvoiceProviderConfig()` 读取的环境变量（`INVOICE_PROVIDER` / `INVOICE_PROVIDER_APP_KEY` / `APP_SECRET` / `API_BASE` / `INVOICE_SELLER_TAX_NO` / `INVOICE_TAX_CODE` / `INVOICE_TAX_RATE` / `INVOICE_PUSH_WECHAT`）**本就是按第三方 SaaS 的 REST 接口形态预留的**，约定返回 `{ invoice_no, invoice_date, invoice_url }`。
- 重要事实：**全电发票（数电票）已同时支持电子普票和电子专票**。即数电专票也是电子化的 —— 这意味着「纸质专票必须邮寄」的场景可大幅减少，仅极少数坚持要纸质票的客户才需走线下，原计划的「纸质专票复用物流模块」负担可降到很低。

## 二、两条路线对比

| 维度 | A. 电子税务局直连（乐企 / 数电平台） | B. 第三方 SaaS（诺诺 / 百望 等） |
|---|---|---|
| 接入门槛 | **高**：需企业资质审批、数字证书、乐企接口报备与联调；小规模/个体可能无乐企资格 | **低**：注册服务商账号、绑定销方税号即可，REST + appKey/secret |
| 与现有桩代码契合度 | **低**：需重写适配层（认证/报文/取版式文件均与现结构不同） | **高**：与现有 `getInvoiceProviderConfig/callInvoiceProvider` 形态一致，几乎不改结构 |
| 开票服务费 | **无**（税局侧免费） | 有：约 0.x ～ 1~2 元/份 或年套餐（以服务商报价为准） |
| 普票 / 数电专票 | 均支持 | 均支持（且封装更顺手） |
| 红冲 / 作废 / 推送微信卡包 | 需自行对接各接口 | **现成接口**，文档/SDK 完善 |
| 数据流向 | 直达税局，**无第三方中转** | 经服务商中转（需评估合规与隐私条款） |
| 稳定性 / 税局规则变更 | 自行跟进维护 | **服务商兜底**，有测试沙箱 |
| 上线速度 | 慢（数周～） | **快（数天级）** |
| 运维成本 | 高（自建+长期维护） | 低 |
| 适合谁 | 开票量大、有专职 IT/财务、要零服务费与数据自主 | 中小开票量、要快速上线、IT 资源有限 |

## 三、推荐：第三方 SaaS（诺诺优先）

针对本项目的判断依据：
1. **改动最小、最快上线**：现有桩代码就是 SaaS 形态，接上即用，不用重构。
2. **开票量级不大**：按维修工单开票，服务费可忽略。
3. **牙科 ToB 需要数电专票**（诊所/医院/经销商进项抵扣），SaaS 对专票/红冲/卡包推送支持成熟。
4. **团队是 uniCloud serverless + 小团队**，不宜自建乐企直连并长期维护。
5. **数电专票电子化**后，基本免除纸质专票寄送负担。

> 何时改选直连：若思科达已具备乐企资格、开票量很大、且有专职财务 IT，则直连可省服务费、数据更自主。

## 四、拍板前必须确认的业务要素（无论选哪条都要）

1. **纳税人身份**：一般纳税人 / 小规模？（决定能否自开专票、适用税率）
2. **税率与税收分类编码**：维修业务到底按哪类？
   - 「修理修配劳务」通常 **13%**（货物性质）；纯「技术/安装服务费」可能 **6%**。**务必由财务确认**，并据此设 `INVOICE_TAX_RATE` 与 `INVOICE_TAX_CODE`。
3. **销方信息**：销方税号 `INVOICE_SELLER_TAX_NO`、开票名称、（专票还需）开户行/账号/地址电话。
4. **是否开数电专票**：若开，客户端需**补采专票要素**（购方地址电话、开户行及账号）—— 目前客户端只采集了普票要素（抬头/税号/邮箱）。
5. **是否推送微信电子发票卡包**（`INVOICE_PUSH_WECHAT`）。

## 五、选定 SaaS 后的落地步骤（我可执行）

1. 在 `callInvoiceProvider()` 内实现：鉴权 → 调开票下单 → 轮询/回调取版式文件 URL → 返回 `{ invoice_no, invoice_date, invoice_url }`。
2. 后台补「发票管理」列表页（批量看待开票/已开票，当前只在工单详情内）。
3. 客户端按是否支持专票，决定是否补采专票要素 + 放开「发票类型」选择。
4. 控制台配置上述 `INVOICE_*` 环境变量，先用服务商沙箱联调，再切生产。
5.（可选）纸质票兜底：仅对坚持要纸质的客户，复用回寄物流模块寄送 —— 量极小。
